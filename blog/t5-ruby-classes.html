A Class Above the Rest

An introduction to Ruby classes, instance variables, and inheritance

The best help in understanding the purpose and importance of object orientated programming is the use of real world examples. It is a big step to go from thinking of a line of code like sum = 2 + 5 or if b > c then … to understanding the creation of an object as an instance of a class that has inherited properties from the classes in the hierarchy above. The example we are going to use today to wrap our heads around Ruby Classes is dogs.

Every dog we see walking outside is unique, but we still instantly recognize it as a dog (except for those tiny lap dogs that might as well be cats or guinea pigs). Although every dog is different, we recognize that they are also all somewhat similar in their “dogness.” In object orientated programming, we define a class to encompass the attributes that all dogs have. If we create a new dog object we first have to identify the basics like it’s breed and how old it is. We then have access to teach that dog a new trick or give it a grooming. In the world of Ruby code, each of these attributes of dogness is handled slightly differently so we will break down each one.

Harper = Dog.new("Goldendoodle")

The first is initializing the new dog object. When we create a new dog we need to identify the dog’s breed. The dog’s breed can’t be changed at any time. That’s just the way our puppy was born and we can’t change it later if we want to. Any type of variable that is stored at the dog level is called an instance variable. This means that the data for that variable is stored within our dog object. In the case of breed, the data for that variable shouldn’t be able to be changed, but should be available to be read later if desired. So we can get the dog breed by:

p Harper.breed
    => "Goldendoodle"

But we CAN NOT set the dog breed by:

Harper.breed = "German Shepard"

We can also have instance variables that can be read or set. For example, Harper’s coat color is white, but if I take her to the groomer I can ask for them to turn her into a zebra for Halloween:

p Harper.coat
    => "white"
        
    Harper.coat = "black and white stripes"
    p Harper.coat
    => "black and white stripes"

How these variables are set up within our Ruby Class is different to account for their different levels of access. For variables that should only be read, but not edited directly by the user we use the code:

attr_reader :breed

For the attributes that are readable and writable we instead use:

attr_accessor :coat

There is also a way to make an instance variable only writable, but not readable. However, I can’t think of anything like that in relation to our dog example. Here’s the line of code that you would use if you had a good reason for that type of data:

attr_writer :example_variable

Dogs are definitely the best pets, but they aren’t the only kind of pets. In fact, they have a lot in common with other pets. If we were building out all of the different classes for each type of pet we would have to use a lot of redundant code. However, Ruby has a way to help with this (of course). The best way to set this up is to first build out a class for Pet. In this class we build out the variables or methods that are useful for any type of pet. Then when we define our Dog class we would tell it to inherit all of the methods and variables from the pet class. In this way we only need to define the things in the dog class that are unique to dogs.

def Pet
      att_reader: breed
      
      def train(new trick)
        commands << new_trick
      end
    end

    def Dog < Pet
      attr_accessor :coat
    end

    Harper = Dog.new
    Harper.train("sit")
